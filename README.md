### 配送プラットフォームのインフラ構築(2025年~2026年)

#### 担当
Google Cloud を使用したインフラの設計・構築を担当しました。
Terraform を用いてゼロからインフラ構築を実施しました。

#### 使用技術
- IaC: Terraform（モジュール構成）
- コンテナ基盤: Cloud Run
- データベース: AlloyDB for PostgreSQL（Primary/Read Pool 構成、IAM 認証）
- ネットワーク: VPC、Cloud Load Balancing、Cloud NAT、Private Service Access（PSA）
- CI/CD: GitHub Actions、Workload Identity Federation
- セキュリティ: Cloud Armor、IAP（Identity-Aware Proxy）
- シークレット: Secrets Manager、SOPS、Cloud KMS
- ドメイン管理: Cloud DNS
- 監視: Cloud Monitor
- ポリシー・品質: Conftest（OPA）、Trivy
- メッセージング: Pub/Sub

#### 主な業務内容
Terraform を用いてインフラ全体をゼロから構築・プロビジョニングしました。以下、主要な業務内容の詳細です。

- **ネットワーク設計・構築**
  - プライベート接続
    - Private Service Access（PSA）を使用して、Cloud Run から AlloyDB へのプライベート接続を確立
    - Cloud Run 同士のプライベート接続を確立。内部アプリケーションロードバランサ・プライベート DNS を設定して、宛先の Cloud Run リソースと通信するように実現
  - 外部向けロードバランシング
    - 外部アプリケーションロードバランサの構築
    - SSL 証明書管理
    - Cloud Armor による IP whitelisting
    - パスベースルーティング設定
  - VPC ネットワーク設計
    - 用途別にサブネット分離

- **Cloud Run から AlloyDB への接続方法を設計・実装**
  - サイドカーコンテナ上の AlloyDB Auth Proxy を経由し、IAM 認証によりパスワードレスでの安全な接続を実現

- **AlloyDB のクラスタ構築**
  - Primary/Read Pool 構成による読み書き分離
  - DB 初期設定（データベース作成・ユーザー権限設定）用のシェルスクリプトを実行する Cloud Run Job を作成し、デプロイフローに組み込み

- **AlloyDB のバックアップと復元の戦略策定と構築**
  - ポイントインタイムリカバリ（PITR）のワークフローの設計。GitHub Actions の workflow_dispatch で以下の2段階で PITR を完遂できるように構築。
    - 復元時間をワークフロー実行時の入力値として、復元用 PR を生成（マージ後に復元クラスタが apply される）
    - 旧クラスタを Terraform 管理下から外し、各種サービスの接続先を復元クラスタへ切り替える PR を生成
      - ワークフロー内で、Terraform の state ファイルを編集する必要があり破壊的変更が生じるため、その前に実施可能な Dry Run モードを用意。ワークフロー実行時のフラグの入力値によって Dry Run モードで実行できるようにし、一時的な state ファイルを作成・参照することで、本番の state ファイルに影響を与えずに plan 結果を確認できるようにした。あわせて Conftest で terraform plan の結果を検証するポリシーを導入し、意図しないリソース変更を防止した。

- **AlloyDB の初期ユーザの DB パスワードローテーションの設計・実装**
  - Secret Manager のローテーション通知を Pub/Sub で受け取り、新パスワードを生成して Secret Manager と AlloyDB の両方に反映する Cloud Run サービスを Go で開発

- **シークレットの管理設計**
  - state ファイル上にシークレットを含めないように、Terraform では Secrets Manager を管理しない方針
  - GitHub Actions の Secret 設定にシークレットを入れない方針。Workflow を push できる人であれば自由に取得できる状況を防止
  - アプリ用のシークレットは SOPS で暗号化したファイルをリポジトリで管理。GCP KMS で復号。
    - main ブランチへの push 時に GitHub Actions のワークフローで以下を実現
      - 復号 → Secret Manager 更新 → 参照元の Cloud Run サービスの再デプロイによりシークレットの更新をサービスに反映

- **GitHub Actions による CI/CD パイプラインの設計構築（Workload Identity 連携）**
  - インフラ構成のデプロイ用 CI/CD
    - terraform plan/apply を実行するそれぞれの Workflow で借用するサービスアカウントを分離
    - feature ブランチでは apply 用サービスアカウントを借用できないように設定
  - アプリケーションのデプロイ用 CI/CD
    - docker build/push までを実現する CI を構築
    - DB Schema Migration（Cloud Run Job で実行）→ Cloud Run へイメージデプロイを実行する CD を構築
  - 共通基盤用の Cloud Run サービス・ジョブの Build and Deploy ワークフローを構築。PR ではラベル付与で検証環境へデプロイ可能

- **Terraform の品質・セキュリティ**
  - Conftest による Policy as Code 基盤の構築。Rego 言語を使用して、HCL ファイルを静的解析して設定をチェックするポリシーを記述。
    - 例: 重要リソースへの `prevent_destroy` 設定のチェック、GCS バケットの均一なバケットレベルアクセス（`uniform_bucket_level_access`）設定のチェック
  - PR の作成・push 時に Conftest/Trivy によるポリシー・セキュリティチェックを実施する GitHub Actions のワークフローを構築。結果を PR コメントで表示。

- **Cloud Monitoring によるモニタリングとアラート通知の実装**
  - Cloud Run サービスの CPU・メモリ使用率を監視し、閾値超過時にアラートを発報する仕組みを構築
  - ログベースのエラー検知アラートを実装し、Cloud Run のエラーログを自動検知
  - AlloyDB の CPU・メモリ使用率アラートを設定（ADR で閾値・対応方針を記載）
  - Slack 通知チャネルを設定し、アラート発生時に詳細情報（プロジェクト・サービス名・エラーメッセージ等）を含む通知を自動送信

- **インフラ構成図の作成**
- **ドキュメンテーション**
  - 設計判断を ADR（Architecture Decision Records）として記録
  - 手順書・ガイドを整備

#### 習得スキル
- Google Cloud/Terraform を用いた本番運用可能なインフラの設計・構築能力
- セキュリティを考慮したシークレット管理戦略の策定と実装知識（SOPS+KMS、Secret Manager、Workload Identity による権限分離）
- データベースのバックアップと復元の運用知識
- 最小権限の原則に基づいた IAM と CI/CD 連携のセキュアな設計（Workload Identity 連携、plan/apply・ブランチ別のサービスアカウント分離）
- インフラ変更の品質担保（Conftest/OPA、Trivy を活用した設定チェック）
- イベント駆動による自動化（Pub/Sub と Cloud Run を組み合わせたシークレットローテーション連携）

### 決済プラットフォームの開発(2022年~2023年)
#### プロジェクト概要
日本最大級の決済プラットフォームの開発

①自チーム構成<br/>
PdM 1人
SM 1人
dev 6人<br/>

②参画時のプロジェクトの状況<br/>
既にサービスが稼働している状態で、新規機能を追加していく段階で参画させていただきました。

③チーム内の自身の役割<br/>
バックエンドの開発を担当しています。

④仕事の進め方<br/>
スクラム開発をベースに進めています。具体的には2週間のイテレーションで開発を進めつつ、スプリントプランニングやレトロスペクティブ、バックログリファインメントなどの各種イベントを実施します。<br/>

#### 担当
- 決済プラットフォームの開発のバックエンドを担当しました。

#### 使用技術
- 開発言語 Go
- コンテナ基盤 Kubernetes(GKE)
- 通信プロトコル gRPC(マイクロサービス間), HTTP+JSON(Web)
- データストア Cloud SQL, Cloud Datastore
- ストレージ Cloud Storage
- CI/CD GitLab CI, ArgoCD
- メッセージング Kafka, Cloud PubSub
- ビルド Makefile
- ログ基盤 Cloud Logging
- 分散トレーシング OpenTelemetry
- 認証/認可 OIDC/JWT, OPA
- IaC Terraform

#### 主な業務内容
- Callback機能の非機能改善。リトライ部分の非同期化を実現するための方針検討と設計を担当。Callbackの順序保証やpod異常終了時のロスト回避を考慮しながら実現。
- Callback失敗時のアラート通知の実現。
- 新MSのクライアント基盤構築
- 非機能試験用の大量データ生成バッチ構築。大量データを効率よく投入する方法を調査・検証しながら模索。GCSバケットに置いたCSVファイルをCloudSQLとBigQueryにインポートする流れをJobで実現。
- Terraformでインフラ構成の編集
- Kubernetesのマニフェストファイルの作成(Helmで管理)
- docker-composeを使用したローカルテスト用の環境構築。テスト前処理等をシェルスクリプトで記述
- テスト改善の策定と実装。テーブル駆動テストによりデータ部と処理部の切り分け。Functional Options Patternでモックの戻り値などを柔軟に共通化や指定可能に実現。
- 各マイクロサービスの不足しているユニットテストを洗い出し実装してカバレッジを向上。GoMockを使用。
- GKEのアップデート時の可用性向上のための改善。Podの削除時に、データの不整合等が発生しないように終了処理をする仕組みを構築。Pod上に起動しているアプリケーションのプロセスにSIGTERMが通知されてから、SIGKILLが発動するまでに、contextやgoroutineの制御などによって正常に終了するように実装。
- protoファイル作成
- 某決済種による取引の検索機能の実装
- BigQueryに保存されている取引情報の検索機能の実装。
- 結合テストのシナリオ策定、ドキュメント作成、実施。結合テストで生じた不具合の修正。

#### 発揮したバリュー
- Callback機能のリトライ部分の非同期化の方針決めと設計を担当しました。Callbackを実施する順番を決められた順序から崩さずに、且つpodが途中で遮断した際に処理中のデータが抱え落ちしないことを保証しながら、非同期化を実現する必要がありました。それらを踏まえて3つの案を出しました。<br>
①ゴルーチン内でのタスクの順序を制御する社内ライブラリを使用する案<br>
②kafkaで新topicを作成してコンシューマー側で非同期でリトライ実施する案<br>
③バッチとしてリトライ部分を切り出して定期実行する案<br>
案ごとにメリットデメリットを列挙したり、アクティビティ図で詳細な設計を表したりして共有しました。実現可能性やパフォーマンスを比較しながら③にすることが決まりました。

#### 習得スキル
- GoとgRPCを使用したマイクロサービス構成での開発
- Kubernetesや周辺ツールへの基本的な理解
- IAMやGCS、CloudSQL、BigQueryなどのGCPサービスを活用した設計・構築
- Terraformによるインフラ構成の理解や編集

言語 C#（ASP.NET Core）<br>
データベース MySQL, Redis<br>
O/Rマッパー EFCore, Dapper<br>
負荷試験ツール JMeter<br>
コンテナ基盤 Docker<br>
テスト NUnit<br>

- 負荷シナリオ作成と試験の実施<br>
JMeter, Nginx(リクエストのばらつき具合をログで確認するため)

#### 業務内容
##### 負荷試験用アプリ開発
- ユーザー新規登録/一覧取得/個別取得/編集/削除の機能実装
- ログイン・ログアウト機能実装(Keyにトークン、valueにuserIdを格納。expireを1分に設定)
- TODOの一覧取得/個別取得/編集/閲覧/削除の機能実装

##### 負荷シナリオ作成
- JMeterという負荷検証ツールを使って、負荷シナリオを作成
 - シナリオテスト用(シナリオが正しいかを検証)
 - ループテスト用(ループを増やしても正しく動くかを検証)
 - スレッドテスト用(複数のスレッドで正しく動くかを検証)
 - ロングランテスト用(30分間負荷をかけても正しく動くかを検証)
- 負荷試験の実行
 - レスポンスタイムを集計するシェルスクリプトの作成
 - 実際に負荷をかけて、メモリの使用量が経過と共に上昇していないかを確認
 - APIごとの平均・最大レスポンスタイムを計測

#### 工夫した点
- 負荷試験で実際に負荷をかけると、各リクエストのタイミングが揃いすぎてしまう問題が発生しました。リクエストを適切にばらけさせるために、jmeterのクライアントとapiサーバとの間にnginxを挟んで、アクセスログを見ながら、ランプアップ時間などを調整して、負荷試験を、実際のシチュエーションに近づけることができました。
- 負荷試験の結果、どのAPIも最大レスポンス時間が1000msを超えていたので、パフォーマンスを改善するために、MySQLの実行プランを確認して、改善できそうなところを特定して、インデックスを貼りました。その結果、各APIごとの最大レスポンスタイムの改善が見られました。

- 負荷試験のシナリオ設計や構築、実施
